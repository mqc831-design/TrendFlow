# 今日热搜 · 需求规格说明书

## 1. 项目概述

### 1.1 产品定义

「今日热搜」是一个多平台热榜聚合网站，将微博、知乎、B 站三个平台的公开热榜数据统一聚合，以卡片网格形式在首页展示，支持公网访问。

### 1.2 产品定位

学习项目 + 作品展示项目。第一版验证「真实数据获取 → 后端清洗缓存 → 前端展示 → 配置能力 → 公网部署」的完整链路。

### 1.3 技术栈

| 层级 | 技术 |
|---|---|
| 前端 | React + TypeScript + Vite |
| 样式 | CSS Modules |
| 后端 | Node.js + Express |
| 数据源 | 开源热榜 API 项目（自部署） |
| 缓存 | 内存 Map |
| 部署 | 前端 Vercel，后端 Railway |

## 3. 功能需求

### 3.1 首页热榜聚合 (FR-HOME)

**FR-HOME-01 多平台卡片展示**

首页以卡片网格布局展示微博、知乎、B 站三个平台的热榜，每个平台一张卡片。

**FR-HOME-02 卡片内容**

每张卡片包含：
- 平台名称（微博 / 知乎 / B 站）
- 该平台数据最后更新时间
- 热榜条目列表，共 Top 20 条

**FR-HOME-03 热榜条目字段**

每条热榜条目展示：
- 排名（1～20）
- 标题（可点击跳转）
- 热度值（无数据时显示「暂无热度」）
- 标签（如「热」「新」「荐」，无数据时隐藏）
- 摘要（无数据时隐藏）

**FR-HOME-04 标题跳转**

点击热榜条目标题，在新标签页打开原平台对应内容页面或搜索页面。

**FR-HOME-05 手动刷新**

首页提供刷新按钮。用户点击后触发后端强制刷新所有平台数据，更新成功后前端展示最新数据；更新失败时返回上一次可用缓存并提示用户。

**FR-HOME-06 最后更新时间**

首页顶部显示全局最后更新时间。各平台卡片也显示各自的数据更新时间。

**FR-HOME-07 页面头部**

显示网站名称「今日热搜」和简短说明「聚合微博、知乎、B 站今日热榜」。

**FR-HOME-08 页脚声明**

页面底部注明「本网站仅用于学习研究，非商业用途」及数据来源说明。

### 3.2 配置界面 (FR-CONFIG)

**FR-CONFIG-01 关键字配置**

用户可以输入关注关键字，多个关键字用逗号分隔。输入后首页热榜中命中关键字的条目标记展示，关键字高亮显示。

**FR-CONFIG-02 关键字匹配逻辑**

支持三种匹配模式：
- `any`：命中任一关键字即标记
- `all`：必须同时命中所有关键字才标记
- `exclude`：排除包含指定关键字的内容

**FR-CONFIG-03 数据源配置**

用户可以勾选启用或停用的平台数据源（微博、知乎、B 站）。

**FR-CONFIG-04 展示数量配置**

每个平台展示条数可配置，可选值：10 / 20 / 30，默认 20。

**FR-CONFIG-05 刷新策略配置**

支持配置：
- 刷新模式（manual / scheduled / both）
- 每日定时刷新时间（默认 08:00）
- 缓存 TTL（默认 300 秒）

**FR-CONFIG-06 关键字命中标记展示**

首页命中关注关键字的条目：
- 标题旁显示「已关注」标签
- 命中的关键字在标题中高亮
- 卡片顶部显示「命中关注关键词 X 条」

首页仍展示完整热榜，不使用强过滤隐藏未命中内容。

### 3.3 缓存与刷新 (FR-CACHE)

**FR-CACHE-01 后端缓存**

后端使用内存 Map 缓存各平台热榜数据，TTL 可配置（默认 300 秒）。

**FR-CACHE-02 缓存失效逻辑**

当请求到达时，优先判断缓存是否有效。有效则直接返回缓存数据；失效则重新请求第三方数据源并更新缓存。

**FR-CACHE-03 失败回退**

若第三方接口请求失败，优先返回上一次可用缓存数据，并在响应中附带错误提示信息。

**FR-CACHE-04 手动刷新**

通过 `/api/refresh` 接口强制绕过 TTL，重新拉取所有启用平台的最新数据并更新缓存。

**FR-CACHE-05 每日定时刷新**

每天 08:00 后端执行一次定时刷新任务（使用 node-cron），确保早上打开页面时已有较新的缓存数据。

### 3.4 公网部署 (FR-DEPLOY)

**FR-DEPLOY-01 前端部署**

前端部署到 Vercel，支持 HTTPS 公网访问。

**FR-DEPLOY-02 后端部署**

后端部署到 Railway，支持 HTTPS 公网访问。

**FR-DEPLOY-03 环境变量管理**

后端通过环境变量管理配置项：
- `PORT`：服务端口（默认 3000）
- `CACHE_TTL_SECONDS`：缓存 TTL（默认 300）
- `DAILY_REFRESH_TIME`：每日刷新时间（默认 08:00）
- 各平台 API 地址

## 5. 数据模型

### 5.1 热榜条目 (HotItem)

```ts
interface HotItem {
  rank: number;           // 排名，1～20
  title: string;          // 标题
  url: string;            // 原平台跳转链接
  heat?: string | null;   // 热度值，可选
  summary?: string;       // 摘要，可选
  tag?: string;           // 标签（热/新/荐），可选
  createdAt: string;      // 数据生成时间 ISO 8601
  updatedAt: string;      // 数据更新时间 ISO 8601
}
```

### 5.2 平台榜单 (HotList)

```ts
interface HotList {
  platform: 'weibo' | 'zhihu' | 'bilibili';
  platformName: string;   // 展示用平台名称
  sourceName: string;     // 数据来源名称
  updatedAt: string;      // 该平台数据更新时间 ISO 8601
  items: HotItem[];       // 最多 20 条
}
```

### 5.3 聚合接口响应 (HotListResponse)

```ts
interface HotListResponse {
  updatedAt: string;      // 全局更新时间 ISO 8601
  cache: {
    hit: boolean;         // 是否命中缓存
    ttl: number;          // 剩余有效时间（秒）
  };
  data: HotList[];        // 各平台榜单数据
}
```

### 5.4 字段展示规则

| 字段 | 展示规则 |
|---|---|
| rank | 必填，展示为 1～20 |
| title | 必填，支持点击跳转 |
| url | 必填，`target="_blank"` 跳转 |
| heat | 可选；无值时显示「暂无热度」 |
| summary | 可选；无值时隐藏 |
| tag | 可选；无值时隐藏 |
| updatedAt | 展示为「更新时间：YYYY-MM-DD HH:mm」 |

## 6. 接口规格

### 6.1 获取聚合热榜

```
GET /api/hot-lists
```

返回微博、知乎、B 站三个平台的聚合热榜数据。响应格式见 5.3 HotListResponse。

### 6.2 获取单平台热榜

```
GET /api/hot-lists/:platform
```

`:platform` 取值：`weibo` | `zhihu` | `bilibili`

返回指定平台的 HotList 对象。

### 6.3 手动刷新

```
POST /api/refresh
```

强制刷新所有启用平台的数据。刷新成功返回最新数据；刷新失败返回上一次缓存数据及错误信息。

### 6.4 读取配置

```
GET /api/config
```

返回当前关键字、数据源、刷新策略等配置。

### 6.5 更新配置

```
POST /api/config
```

更新配置项。第一版不做登录鉴权，需在后端限制写入操作或仅用于本地调试。

## 7. 非功能需求

### 7.1 性能

| 指标 | 要求 |
|---|---|
| 首页加载时间 | ≤ 3 秒（优先返回缓存） |
| 缓存命中率 | 正常运行时 ≥ 90% |

### 7.2 可用性

| 指标 | 要求 |
|---|---|
| 第三方接口失败 | 返回上一次可用缓存，不返回空白页 |
| 服务重启后 | 首次请求自动拉取新数据 |

### 7.3 可维护性

- 每个平台的数据获取和清洗逻辑封装为独立适配器模块
- 新增平台时，只需新增适配器文件，不修改核心逻辑

### 7.4 安全性

- 不收集和存储用户个人信息
- 第一版不开放公网配置写入（配置接口仅本地使用或 Token 保护）
- 后端不对用户输入做持久化存储

### 7.5 体验

- 页面响应式布局，移动端可正常阅读
- 加载中显示骨架屏或 Loading 状态
- 接口异常时显示有好的错误提示

## 8. 数据源适配器

后端为每个平台实现独立的数据适配器模块，格式如下：

```
src/adapters/
  weibo.adapter.js    # 微博热搜适配器
  zhihu.adapter.js    # 知乎热榜适配器
  bilibili.adapter.js # B站热榜适配器
```

每个适配器暴露统一接口：

```ts
interface Adapter {
  platform: string;
  fetchHotList(): Promise<HotList>;
}
```

适配器负责：
1. 调用对应的开源热榜 API
2. 将原始响应数据清洗为 HotList 标准格式
3. 处理请求异常（抛出统一错误格式）
