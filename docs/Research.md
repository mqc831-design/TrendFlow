# 今日热搜 · 产品调研文档

## 1. 项目背景

“今日热搜”是一个多平台热榜聚合网站，目标是将微博、知乎、B 站等平台的公开热榜数据进行统一聚合，并以简洁的卡片网格形式展示在首页，方便用户快速浏览不同平台的热点内容。

本项目定位为**学习项目 + 作品展示项目**。第一阶段重点验证从第三方 JSON 数据接口获取热榜数据、后端统一中转、前端卡片化展示、缓存刷新和公网部署的完整链路，为后续继续扩展为更完整的信息聚合、关键词监控或热点分析工具打基础。

---

## 2. 项目目标

### 2.1 核心目标

建设一个可公网访问的多平台热搜聚合网站，首页展示微博、知乎、B 站三个平台的今日热榜，每个平台以独立卡片呈现，用户可以通过链接访问网站并查看最新热榜内容。

### 2.2 产品价值

| 价值方向 | 说明 |
|---|---|
| 学习价值 | 覆盖前端展示、后端接口、第三方 API 调用、缓存、部署等完整 Web 项目链路 |
| 展示价值 | 可作为作品集项目，对外展示产品设计、工程实现和部署能力 |
| 使用价值 | 快速查看多个平台热榜，减少在不同 App / 网站之间切换的成本 |
| 扩展价值 | 后续可扩展关键词配置、数据源配置、趋势分析、订阅提醒等能力 |

## 3. 用户与使用场景

### 3.1 目标用户

| 用户类型 | 使用诉求 |
|---|---|
| 项目作者本人 | 学习完整 Web 项目开发流程，沉淀作品展示材料 |
| 展示对象 | 通过公网链接直观看到项目效果，了解项目完整度 |
| 普通访问者 | 快速浏览微博、知乎、B 站的实时热点内容 |

### 3.2 典型场景

1. 用户打开首页，直接看到微博、知乎、B 站三个热榜卡片。
2. 用户查看每个平台 Top 20 热榜内容。
3. 用户点击某条热搜，跳转到原平台页面查看详情。
4. 用户点击刷新按钮，触发后端重新拉取数据或返回当前有效缓存。
5. 用户进入配置界面，配置关注关键字、查询逻辑和数据源策略。
6. 系统每天 08:00 自动刷新一次热榜数据，保证首页具备基础时效性。

## 4. 调研发现

### 4.1 热榜聚合类产品的常见形态

当前常见的热榜聚合类网站通常采用“多平台卡片聚合”的首页结构，即一个平台对应一个榜单卡片，每张卡片展示平台名称、更新时间、榜单条目和跳转链接。

常见展示字段包括：

| 字段 | 说明 |
|---|---|
| 排名 | 当前热榜中的序号 |
| 标题 | 热搜或热榜条目的标题 |
| 链接 | 跳转到原平台内容页或搜索页 |
| 热度 | 部分平台提供热度值、讨论量或浏览量 |
| 标签 | 部分平台提供分类、热、新、荐等标记 |
| 摘要 | 部分平台提供内容简介，适合做补充展示 |

### 4.2 数据获取方式调研

由于浏览器直接请求第三方平台接口通常会遇到跨域限制、鉴权限制或反爬限制，因此第一版应采用**后端中转**模式：

```text
前端页面 → 自有后端 API → 第三方热榜 JSON 接口 → 数据清洗 → 缓存 → 返回前端
```

第一版不建议直接做 HTML 页面爬虫，优先使用可返回 JSON 的第三方热榜接口或开源聚合 API。这样实现成本更低，也更适合作为学习项目快速落地。

### 4.3 可参考的数据源方向

| 数据源方向 | 说明 | 适用性 |
|---|---|---|
| 第三方热榜聚合 API | 已整合多个平台热榜，调用简单，适合 MVP | 高 |
| 开源热榜 API 项目 | 可自部署，便于控制稳定性和学习后端实现 | 高 |
| 各平台非官方 JSON 接口 | 数据更贴近源站，但稳定性和兼容性需要验证 | 中 |
| HTML 爬虫 | 灵活度高，但维护成本、风险和复杂度更高 | 低，第一版不建议 |

第一版建议优先选择第三方 JSON API 或开源热榜 API 项目，后端统一封装为自有接口，避免前端直接绑定第三方接口结构。

## 6. 页面与功能设计

## 6.1 首页：今日热搜合集

首页是项目的核心展示页，采用简洁风格，重点突出内容本身。

### 页面结构

```text
顶部区域
- 网站名称：今日热搜
- 简短说明：聚合微博、知乎、B站今日热榜
- 最后更新时间
- 手动刷新按钮

主体区域
- 微博热搜卡片
- 知乎热榜卡片
- B站热榜卡片

底部区域
- 非商业用途说明
- 数据来源说明
```

### 卡片内容

每个平台卡片包含：

| 元素 | 说明 |
|---|---|
| 平台名称 | 如微博、知乎、B 站 |
| 更新时间 | 当前平台数据的更新时间 |
| 榜单列表 | 展示 Top 20 |
| 热榜条目 | 排名、标题、热度、标签、摘要 |
| 跳转链接 | 点击标题跳转原平台页面 |

### 交互逻辑

1. 页面首次加载时，请求后端聚合接口。
2. 后端优先判断缓存是否有效。
3. 缓存有效时，直接返回缓存数据。
4. 缓存失效时，重新请求第三方数据源。
5. 用户点击“刷新”按钮时，触发一次主动刷新。
6. 若第三方接口失败，优先返回上一次可用缓存，并提示数据可能不是最新。

## 6.2 配置界面

配置界面用于体现项目的可扩展性，也是作品展示时的重要亮点。第一版配置可先保存在前端状态或简单配置文件中，不强制做数据库持久化。

### 配置目标

支持用户配置：

1. 关注关键字。
2. 查询逻辑。
3. 启用或停用的数据源。
4. 每个平台展示数量。
5. 刷新策略。

### 配置项建议

| 配置项 | 类型 | 示例 | 说明 |
|---|---|---|---|
| keywords | string[] | AI、安全、股票、云计算 | 用于标记或过滤关注内容 |
| matchMode | enum | any / all / exclude | 关键字匹配逻辑 |
| sources | object[] | weibo / zhihu / bilibili | 配置启用的数据源 |
| topN | number | 20 | 每个平台展示条数 |
| refreshMode | enum | manual / scheduled / both | 刷新模式 |
| scheduledTime | string | 08:00 | 每日定时刷新时间 |
| cacheTTL | number | 300 / 600 | 缓存有效期，单位秒 |

### 关键字匹配逻辑

| 逻辑 | 说明 |
|---|---|
| any | 命中任一关键字即可标记为关注 |
| all | 必须同时命中所有关键字 |
| exclude | 排除包含指定关键字的内容 |

第一版建议先做“高亮标记”而不是“强过滤”。这样首页仍展示完整热榜，同时将命中关键字的条目标记出来，展示效果更直观。

## 7. 数据字段设计

### 7.1 标准热榜条目字段

```ts
interface HotItem {
  rank: number;
  title: string;
  url: string;
  heat?: string | number | null;
  summary?: string;
  tag?: string;
  createdAt: string;
  updatedAt: string;
}
```

### 7.2 平台榜单字段

```ts
interface HotList {
  platform: 'weibo' | 'zhihu' | 'bilibili';
  platformName: string;
  sourceName: string;
  updatedAt: string;
  items: HotItem[];
}
```

### 7.3 聚合接口返回字段

```ts
interface HotListResponse {
  updatedAt: string;
  cache: {
    hit: boolean;
    ttl: number;
  };
  data: HotList[];
}
```

### 7.4 字段展示规则

| 字段 | 展示规则 |
|---|---|
| rank | 必填，展示为 1～20 |
| title | 必填，支持点击跳转 |
| url | 必填，跳转原平台 |
| heat | 可选；无数据时显示“暂无热度” |
| summary | 可选；没有摘要时隐藏 |
| tag | 可选；没有标签时隐藏 |
| createdAt | 用于记录数据生成时间，一般不直接展示 |
| updatedAt | 展示为平台更新时间或页面整体更新时间 |

## 8. 技术方案调研

## 8.1 技术栈

| 层级 | 技术选择 | 说明 |
|---|---|---|
| 前端 | React + TypeScript + Vite | 轻量、适合学习项目和作品展示 |

## 8.2 前后端分工

| 模块 | 前端职责 | 后端职责 |
|---|---|---|
| 首页展示 | 卡片布局、列表渲染、刷新交互 | 提供统一聚合数据接口 |

| 错误处理 | 展示友好提示 | 捕获接口异常、返回缓存或错误信息 |

## 9. 后端接口设计建议

### 9.1 聚合热榜接口

```http
GET /api/hot-lists
```

说明：返回微博、知乎、B 站三个平台的聚合热榜。

### 9.2 单平台热榜接口

```http
GET /api/hot-lists/:platform
```

示例：

```http
GET /api/hot-lists/weibo
GET /api/hot-lists/zhihu
GET /api/hot-lists/bilibili
```

说明：返回指定平台热榜，便于调试和后续扩展。

### 9.3 手动刷新接口

```http
POST /api/refresh
```

说明：主动刷新所有启用平台的数据。

建议逻辑：

1. 用户点击前端刷新按钮。
2. 前端调用 `/api/refresh`。
3. 后端绕过 TTL 或强制更新缓存。
4. 更新成功后返回最新数据。
5. 更新失败时返回上一次缓存，并附带错误提示。

### 9.4 配置读取接口

```http
GET /api/config
```

说明：读取当前关键字、数据源、刷新策略等配置。

### 9.5 配置更新接口

```http
POST /api/config
```

说明：更新配置。第一版如不做登录，需要谨慎开放公网写入能力。作品展示场景下，建议先只做本地配置或前端模拟配置，避免公网用户随意修改。

## 10. 缓存与刷新策略

### 10.1 缓存策略

第一版采用后端内存 Map 缓存。

| 项目 | 设计 |
|---|---|
| 缓存位置 | Node.js 服务内存 |
| 缓存结构 | Map<platform, HotList> |
| TTL | 300～600 秒 |
| 失效逻辑 | 超过 TTL 后重新拉取 |
| 失败回退 | 第三方接口失败时返回上一次可用缓存 |

### 10.2 刷新策略

| 刷新方式 | 说明 |
|---|---|
| 页面首次加载 | 自动请求聚合接口 |
| 手动刷新 | 用户点击刷新按钮触发 |
| TTL 自动刷新 | 缓存过期后再次请求时刷新 |
| 每日定时刷新 | 每天 08:00 后端主动刷新一次 |

### 10.3 每日定时刷新说明

每天 08:00 执行一次定时刷新任务，用于保证早上打开页面时已有一份较新的热榜缓存。

第一版可使用 Node.js 定时任务方案，例如 `node-cron`。如果部署平台存在休眠机制，需要注意定时任务可能不会稳定运行。对于 Railway 等平台，应结合实际部署计划验证服务是否会休眠。

## 11. 配置能力设计

配置能力是本项目区别于普通热榜展示页的重要扩展点。它的价值在于让用户不只是“看热榜”，还能围绕自己关注的方向做内容筛选和标记。

### 11.1 配置界面第一版建议

第一版配置界面不必做得复杂，重点展示以下能力：

| 配置模块 | 第一版做法 |
|---|---|
| 关键字配置 | 输入多个关键字，用逗号分隔 |
| 查询逻辑 | 支持任一命中、全部命中、排除关键字 |
| 数据源选择 | 勾选微博、知乎、B 站 |
| 展示数量 | 默认 20，可配置为 10 / 20 / 30 |
| 刷新配置 | 展示当前 TTL 和每日刷新时间 |

### 11.2 关键字命中展示

首页热榜条目命中关键字后，可进行轻量标记：

1. 标题旁展示“已关注”标签。
2. 命中的关键字高亮。
3. 卡片顶部显示“命中关注关键词 X 条”。

第一版建议不要默认隐藏未命中的内容，避免用户误以为热榜数据不完整。

## 13. 风险与约束

| 风险 | 说明 | 应对方式 |
|---|---|---|
| 第三方接口不稳定 | 免费或非官方接口可能变更、限流、下线 | 后端封装适配层，便于替换数据源 |
| 数据字段不一致 | 不同平台字段名称和结构不同 | 统一清洗为标准 HotItem 结构 |
| 跨域限制 | 浏览器直接请求第三方接口可能失败 | 通过自有后端中转请求 |
| 部署平台休眠 | 免费后端平台可能休眠，影响定时任务 | 保留首次访问刷新逻辑；后续可改用外部定时触发 |
| 缓存丢失 | 内存缓存随服务重启丢失 | 第一版可接受；后续可引入 Redis 或文件缓存 |
| 合规风险 | 热榜数据来源于公开网络，但仍需避免商业化滥用 | 页脚声明学习研究、非商业用途 |
| 维护成本 | 上游接口结构变化后需要适配 | 数据源适配器模块化设计 |

## 17. 当前结论

第一版“今日热搜”项目应优先做成一个简洁、可展示、可公网访问的多平台热榜聚合网站。

当前最合适的实现路线是：

1. 前端使用 React + TypeScript + Vite 实现简洁卡片网格首页。
2. 后端使用 Node.js + Express 统一中转第三方热榜 JSON 接口。
3. 数据源优先选择第三方热榜 API 或开源热榜 API 项目，不做 HTML 爬虫。
4. 后端通过内存 Map 做 300～600 秒缓存，并支持失败时返回上一次可用数据。
5. 首页展示微博、知乎、B 站三个平台，每个平台 Top 20。
6. 增加配置界面，用于配置关键字、查询逻辑和数据源，体现项目扩展能力。
7. 前端部署到 Vercel，后端部署到 Railway，满足 HTTPS 公网访问和作品展示需求。

该项目第一版不追求复杂功能，重点是把“真实数据获取 → 后端清洗缓存 → 前端展示 → 配置能力 → 公网部署”这条主链路跑通。后续可在此基础上逐步扩展为关键词监控、热点趋势分析和 AI 热榜摘要工具。
